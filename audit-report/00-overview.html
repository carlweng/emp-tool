emp-tool — repository overview

emp-tool is the foundational substrate of the emp-toolkit: SIMD block types, fast AES / PRG / PRP / hash / GF(2¹²⁸) and elliptic-curve crypto kernels, IO channels, and a boolean-circuit layer of context-bound typed values (Bit / BitVec / UInt / Int / Float over a context) with a compile-once / run-on-any-context frontend and no global backend. It is a library, not a standalone protocol — concrete protocol security lives in the downstream contexts (semi-honest 2PC, malicious 2PC, ZK) that build on it.

1 · What emp-tool is

Mechanically the repo is small and almost entirely header-only: 65 first-party header files (~7,333 lines) plus exactly three compiled translation units — runtime/crypto/ec.cpp, ir/assets.cpp, ir/builtins.cpp. The only large file in the tree is the vendored third_party/sse2neon.h (~11,200 lines) that lets the entire _mm_* intrinsic surface run on arm64.

Conceptually it is three layers with a strict dependency order circuits → ir → runtime, each with its own umbrella header. #include <emp-tool/emp-tool.h> is the full public umbrella over all three; code that needs one layer includes that layer's umbrella:

  • runtime — runtime/{core,crypto,io,execution}/ (umbrella runtime/runtime.h): the SIMD block type, the symmetric/asymmetric crypto kernels, the network channels, and the garbled-circuit leaf primitives (half_gate / privacy_free).
  • ir — the context-free Boolean IR plus ir/context/, ir/session/, and the new ir/wire_value.h (umbrella ir/ir.h): the C++20 BooleanContext concept and the reusable contexts, a protocol-neutral circuit IR with a native .empbc format, the generic WireValue concept, and the Session / DirectSession / SessionIO contracts.
  • circuits — the typed value families, circuits/crypto/, and circuits/frontend/ (umbrella circuits/circuits.h): the *_T<Ctx> value family written against BooleanContext, the in-circuit crypto, and the compile-once / run-anywhere frontend.

2 · Architecture & the layer stack

The include graph forms a clean bottom-up stack with a strict dependency order circuits → ir → runtime. The two design roots — ir/context/concept.h (the BooleanContext concept) and ir/program.h (the BooleanProgram IR) — are pure leaves with zero internal includes, deliberately usable without any backend. Everything typed is written against them.

The central layering invariant

Every circuits/*_T<Ctx> arithmetic value (bit.h, unsigned_int.h, signed_int.h, bitvec.h) depends only on ir/context/concept.h + runtime/core/utils.h (plus numeric_kernels.h and the thin ir/context/checks.h guard, which itself pulls in only runtime/core/utils.h) — no context implementation, no crypto backend, no IO. The one exception is the float type: float.h includes ir/builtins.h + ir/execute.h (and its float_traits.h helper includes ir/builtins.h), because float ops are replayed from prebuilt .empbc artifacts rather than open-coded.

4 · The BooleanContext model — compile once, run anywhere

This is the architectural heart of the repo. There is no global gate backend: every typed value carries a pointer to its Ctx and forwards each gate to it. The C++20 concept requires just four value-return operations over a cheap, copyable wire handle:

concept BooleanContext = std::regular<Ctx::Wire> && requires {
    c.public_bit(bool) -> Wire;   // the only "input": a public constant
    c.and_gate(a,b)    -> Wire;   // the only COSTED gate
    c.xor_gate(a,b)    -> Wire;   // free
    c.not_gate(a)      -> Wire;   // free
};   // OR, equality, mux, arithmetic, crypto, float are all built on these four

The same source therefore runs unchanged over five realizations of the concept, plus any downstream protocol context:

ContextWireMeaning
ClearCtxuint8_tplaintext evaluation (gate counts match a protocol run exactly)
CountCtxuint8_tgate counting
DigestCtxuint32_tdeterministic fingerprint of the gate stream
RecordCtxuint32_trecords an IR BooleanProgram
downstream backendblockSH2PCCtx (garbled 2PC), ZK, malicious 2PC, …

A user writes a pure circuit body (inputs are arguments, the output is the return value — structurally incapable of I/O because it is recorded through RecordCtx), compile()s it once into a context-free Circuit (a validated program + signature), then run()s it on any context — the same replay path the shipped .empbc builtins use. A value type satisfies the generic WireValue concept (ir/wire_value.h) — context-free, fixed-width, with a clear codec — so circuit code names no protocol. I/O is a session concern, not a circuit concern: a DirectSession exposes its gate context via direct_ctx() and feeds inputs / reveals outputs (SessionIO<S,V>, generic over any WireValue, so a session names no value family — adding a value family needs no session edit), with reveal returning std::optional<clear_t> (present on the party that learns it, nullopt otherwise). The base Session concept requires only party(); ClearSession is the plaintext reference.

Why this matters for an audit

Because the gate stream is identical across every context, the plaintext ClearCtx run, the recorded IR, and the cryptographic protocol run are provably the same circuit — and the test suite exploits this to cross-check protocol kernels against plaintext references bit-for-bit. Numeric semantics are explicit and match hardware: UInt_T wraps mod 2ᴺ, Int_T is two's-complement (no signed-overflow UB), Float_T is IEEE binary16/32/64. Full detail in Report 2 and Report 3.

5 · Two crypto worlds

An easy point of confusion: emp-tool has two completely independent crypto subtrees with no shared include path. Keeping them apart is essential when reading the code.

runtime/crypto/ raw-block crypto

The runtime primitives that operate on 128-bit block registers: AES-NI / VAES, fixed-key PRP, AES-CTR PRG, CCRH & MITCCRH correlation-robust hashes, GF(2¹²⁸) arithmetic, OpenSSL SHA-256 & P-256. These power the PRG, OT extension, and garbling backends.

Report: 1 (Runtime)

circuits/crypto/ in-circuit crypto

The same algorithms expressed as boolean circuits over Bit/BitVec values: AES-128 (Boyar–Peralta 32-AND S-box, 6,400 ANDs/block), SHA-256 (24,744 ANDs), Keccak/SHA3-256 (38,400 ANDs). These are what you garble or prove, not what you run natively. Built on circuits/bitvec.h + unsigned_int.h — never on runtime/crypto/.

Report: 3 (Circuits)

Circuit assets are committed, not generated

The 63 .empbc files are pre-generated and git-tracked under emp-tool/ir/files/ — the build does not produce them. CMake's configure_file generates only the search-path header ir/circuit_asset_dirs.h (build dir + install dir); at runtime ir/assets.cpp prepends the $EMP_CIRCUIT_DIR env override ahead of both. The fp32/fp64 assets were generated from modified Berkeley SoftFloat 3e. The tree intentionally records provenance and the runtime ABI, but does not claim a complete reproducible generation harness; see docs/floating_point_circuits.md.

8 · Ecosystem & downstream position

emp-tool is the shared bottom layer of the whole emp-toolkit. Downstream protocol libraries realize their execution backends as BooleanContexts over this substrate, so a circuit that runs on ClearCtx runs on a protocol context unchanged.

9 · Security posture at a glance

emp-tool's job is to be a correct, fast substrate; protocol-level security is the downstream context's responsibility. The cross-cutting items below recur across the detailed reports and are the first things an auditor should internalize.

Posture itemWhat to know
Compile-time-only ISA dispatchThe SIMD tier (VAES/VPCLMULQDQ/AVX-512) is chosen from feature macros at build time — there is no runtime CPUID fallback. A -march=native binary is host-CPU-locked and will SIGILL on an older CPU. Mitigation: EMP_TOOL_NATIVE_ARCH=OFF.
Validation that survives NDEBUG — and that doesn't.empbc structural validation (validate_program) is always-on in Release and reports through the stack-wide fatal error() path. Developer-misuse guards remain debug-only by default: the cross-context guard (check_same_context, opt back in with -DEMP_CONTEXT_CHECKS=1) and the schedule-plan digest staleness check. Peer-controlled EC point lengths are now hard-checked in recv_pt; local misuse checks such as FS double-enable and alignment remain assertions.
Test-mode randomness kill switchEMP_TEST_MODE=1 (or set_test_mode(true)) swaps PRG seeding and EC scalar generation for a deterministic counter stream (seed 0xC0FFEE12345). Intended for wire-byte equivalence testing; an accidentally-set env var makes production randomness fully deterministic.
Asset loading: structurally safe, not authenticatedThe .empbc parser is hostile-input safe (fixed little-endian reads, overflow-checked size math, validate_program) — but there is no signature/MAC. A well-formed-but-malicious circuit at an attacker-controlled $EMP_CIRCUIT_DIR (first in the search order) would load and replay.
Fail-fast by _Exit(1)runtime/core/utils.hpp error() terminates the whole process rather than throwing — so a hostile peer can reliably crash it (no read timeouts → it can also hang it). The IR validator, .empbc loader, and NetIO/TLSIO setup paths follow the same fatal model; rejection tests assert child-process death, not a catchable exception.
Little-endian onlyWire formats and byte/bit packings assume little-endian; a static_assert refuses big-endian builds rather than silently mismatching a peer.
Strong correctness baseline

Symmetric kernels are validated against FIPS-197 KAT / OpenSSL; gfmul against a bit-level reference and field axioms; hash-to-curve against RFC 9380 vectors; in-circuit AES/SHA/Keccak against FIPS/NIST KAT; and test_builtin_circuits.cpp proves the native kernels and the recorded IR builtins agree bit-for-bit. The residual audit surface is the committed .empbc float assets and their off-tree generation provenance, the NDEBUG behaviour of the debug-only developer guards, and the documented UB preconditions (div-by-zero saturation, INT_MIN/−1, default-constructed values).

11 · Repository caveats an auditor will notice

  • Root data dumps, not source. The repo root holds sort_file.h (~17.6 MB, ~640k lines) and sort.txt (~18 MB) — generated data artifacts, not hand-written code. Don't mistake them for the ~7.3K-line first-party source.
  • Stale generated header. build/emp-tool/circuits/circuit_asset_dirs.h is a leftover from an earlier branch layout (it references a non-existent circuits/files path); the live generated header is build/emp-tool/ir/circuit_asset_dirs.h.
  • Working tree ahead of HEAD. The runtime/ir/circuits three-layer refactor (and the session/WireValue redesign these reports describe) is currently uncommitted on main — HEAD is still 7f21adc. The reports describe the working tree on disk, not the committed tree.
  • Pre-alpha API. This development branch's CMake version reads 1.0.0 but names and headers may still move before the first tagged alpha; anything pinned to a release should track v0.3.x instead.
  • One full public umbrella. #include <emp-tool/emp-tool.h> pulls in all three layers (runtime/runtime.h + ir/ir.h + circuits/circuits.h); code that needs one layer includes that layer's umbrella directly.
