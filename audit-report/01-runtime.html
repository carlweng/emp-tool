1 · Role in the stack & the two crypto worlds

Everything in this report sits below the IR and value layers. The dependency order inside the runtime is itself bottom-up: runtime/core (the block hub) → runtime/crypto (symmetric + asymmetric kernels) → runtime/io (channels embed hash/PRG/EC) → runtime/execution (garbling leaf primitives over core + crypto). The umbrella is runtime/runtime.h, which also carries the little-endian static_assert.

An easy point of confusion: emp-tool has two completely independent crypto subtrees with no shared include path. This report is the first one only.

runtime/crypto/ raw-block crypto (this report)

Primitives over 128-bit block registers: AES-NI / VAES, fixed-key PRP, AES-CTR PRG, CCRH & MITCCRH correlation-robust hashes, GF(2¹²⁸) arithmetic, OpenSSL SHA-256 & P-256. These power the PRG, OT extension, and garbling backends.

circuits/crypto/ in-circuit crypto (Report 3)

The same algorithms expressed as boolean circuits over Bit/BitVec values (AES-128 with the Boyar–Peralta 32-AND S-box, SHA-256, Keccak). These are what you garble or prove, not what you run natively. Built on the value layer — never on runtime/crypto/.

Every primitive in Part B operates directly on the 128-bit SIMD register type block. It is not an opaque struct — it is the hardware vector (core/block.h:16 using block = __m128i;). On aarch64 the bundled sse2neon.h shim (block.h:4-8) maps the whole _mm_* surface to NEON. There is no abstraction boundary between "a value" and "a vector register."

2 · A · block — the 128-bit value type

core/block.h core/block.hpp

block is the universal 128-bit word: PRG output, AES blocks, GC labels, OT messages, hash truncations, bit-matrix rows. No block256/block512 type exists — wider SIMD appears only as internal vec_t members of the Lane traits; everything public is 128 bits.

Construction & constants (block.hpp:25-42)

makeBlock(high, low) → block{(long long)low, (long long)high}   // low = lane 0, constexpr
zero_block, all_one_block, select_mask[2] = {zero_block, all_one_block}   // branch-free 0 / all-ones mux mask

makeBlock is inline constexpr so file-scope block constants compile-time evaluate into .rodata (see docs/static_init.md for why that matters to static-init order).

Byte-order trap when reading traces by eye

makeBlock(hi, lo) stores lo first in memory, but operator<< prints data[1] (high) then data[0] (low) — i.e. big-endian display of a little-endian memory layout (block.hpp:52-62). A known source of confusion when diffing wire traces. The stream operator correctly saves/restores flags and fill.

Bit accessors & σ

  • getLSB(x) reads bit 0 of lane 0 — the GC/OT point-and-permute "color" bit (block.hpp:13-15).
  • set_bit(a,i) is set-only (OR), not assign — it cannot clear a bit; the name suggests "write bit i" but means "force bit i to 1" (block.hpp:44-50).
  • sigma(a) = the linear orthomorphism σ((H,L)) = (H⊕L, H) of eprint 2019/074 (block.hpp:36-38: _mm_shuffle_epi32(a, 78) ^ (a & makeBlock(~0,0))). Purely linear (no secret), so not itself a security boundary, but a bug here would break the correlation-robustness argument of every fixed-key-AES caller.

Bulk XOR / compare / transpose

The xorBlocks_arr family is __restrict__ loops — non-overlap is load-bearing (block.h:30-42); aliasing is UB except via the dedicated xorBlocksTo_arr. cmpBlock is deliberately not early-exit (accumulator-OR for ILP on the equal path, block.hpp:88-94):

acc = OR over (x[i] ^ y[i]);  return _mm_testz_si128(acc, acc);
cmpBlock is "constant-ish", not constant-time

It is data-independent in iteration count, but it is not a side-channel-hardened compare in the cryptographic sense. The toolkit does not claim it is safe for comparing secrets against attacker-chosen values under a timing adversary; it is used where both operands are typically known to one party.

sse_trans / sse_trans_n128 (block.hpp:104-277) is the IKNP/OT-extension bit-matrix transpose (nrows,ncols multiples of 8, with a 16×128 in-register fast path). Pure data movement, but intricate pointer arithmetic.

Test gap worth noting

sse_trans is tested only transpose-twice-is-identity (test_block.cpp), not against an independent reference transpose — a transpose bug that is its own inverse would pass. Worth a closer look if OT-extension correctness is in scope.

3 · A · block_vector, simd_tier, constants, test_mode, utils

core/block_vector.h core/constants.h core/test_mode.h core/utils.{h,hpp}

default-init allocator

default_init_allocator<T> overrides construct to default-init (::new(ptr) U;) instead of value-init, so for trivial T it skips zeroing (block_vector.h:15-44). BlockVec = default_init_vector<block> therefore returns uninitialized memory on resize(N)/BlockVec(N).

Uninitialized-memory-to-wire footgun

Reading a BlockVec element before writing it reads stack/heap garbage. Intentional for performance (buffer about to be overwritten by PRG/AES), but a classic source of (a) non-determinism that breaks the test-mode wire-trace guarantee, and (b) potential information leakage if uninitialized blocks reach the wire. Auditors should flag any BlockVec/default_init_vector whose elements can be sent or hashed without being fully written first. (The send_bool tail-byte handling — §C — shows the team is aware of this bug class.)

constants (constants.h:8-24)

constexpr int64_t AES_BATCH_SIZE = 64;                       // :8
const static int NETWORK_STAGING_BUFFER_SIZE = 1024*32;      // :9  (32 KiB)
const static int NETWORK_STREAM_BUFFER_SIZE  = 1024*1024;    // :10 (1 MiB)
const static size_t IO_BOOL_CHUNK_SIZE = 8*1024;             // :14
enum Party : int { XOR = -1, PUBLIC = 0, ALICE = 1, BOB = 2 };  // :20
const static uint32_t MAX_POINT_BYTES = 2048;                // :24

The Party sign matters: XOR = -1 means "return/leave the value as per-party XOR shares rather than the clear value" (each party keeps its own share — not the same thing as a nullopt reveal), versus PUBLIC = 0 ("reveal to both"). Mixing them up is a reveal-policy bug with confidentiality impact. MAX_POINT_BYTES caps the length field for EC point deserialization.

Test mode is a one-env-var randomness kill switch

test_mode.h exposes a global atomic flag seeded from $EMP_TEST_MODE plus a monotonic seed counter starting at 0xC0FFEE12345 (test_mode.h:24-39). When on, PRG() default construction draws a deterministic counter seed and ECGroup::rand_scalar() becomes a PRG-driven sampler. A production binary that ever sets EMP_TEST_MODE=1 (env var or set_test_mode(true)) becomes completely insecure — seeds become 0xC0FFEE12345, +1, +2, …. Only v[0] == '1' is checked (test_mode.h:27-28), so =1 / =10 trigger it; a single env var away from a catastrophic randomness downgrade. Deserves operational guardrails (or a build-time hard-disable in release).

utils — bit/byte packing (the footgun-rich part)

  • error(s) is [[noreturn]], prints to stderr and std::_Exit(1), capturing the call site via __builtin_LINE/FILE default arguments (utils.hpp:9-12, declared utils.h:22-24). The whole library's failure model is fail-fast.
  • bytes_to_bits32 / bits32_to_bytes (SSE/AVX2 paths, utils.hpp:19-67) pack/unpack 32 bools↔bits; bytes_to_bits32 treats any nonzero byte as 1 and reads a full 32 bytes via an unaligned load — callers must ensure 32 bytes are readable.
  • bools_to_bits(out, bools, len) has tail-preserve semantics: the final partial byte is read-modify-written, preserving its high bits (utils.h:41-46, utils.hpp:69-80). Every caller that emits/hashes the packed bytes must zero the tail byte first or use only ⌈len/8⌉ bytes where the tail bits are don't-cares. bool_to_block is safe (128 is a multiple of 32 → whole block overwritten).

4 · B · AES-128 key schedule & ECB

crypto/aes.h crypto/aes.hpp

AES-128-ECB is used in two distinct modes: as a secret-key PRF/PRG and as a fixed/public-key random permutation (basis of fixed-key garbling, JustGarble eprint 2013/426, and the CCRH/MITCCRH hashes). The header cites "Fast Garbling of Circuits Under Standard Assumptions" (eprint 2015/751, aes.h:16-18) for the interleaved multi-key schedule — garbling schedules many one-block keys at once.

Key data structure (aes.h:10)

typedef struct { block rd_key[11]; } AES_KEY;

Exactly 11 round keys × 16 bytes = 176 bytes (AES-128 has Nr=10 rounds → 11 round keys). rd_key[0] is the user key verbatim (aes.hpp:362). There is no decryption schedule — emp only ever encrypts (ECB / CTR / Davies–Meyer).

Alignment is asserted, not enforced

EMP_AES_ASSERT_ALIGNED(p) (aes.h:12-14) asserts 16-byte alignment of round-key / block pointers at the public entry points. It is an assert, so it is compiled out under NDEBUG. In practice the lane loads/stores use unaligned intrinsics (_mm_loadu_si128 / storeu, simd_tier.h:89-90), so a misaligned pointer still works on x86 — but the stated contract is not guaranteed in release.

Key-schedule algorithm (aes.hpp:352-371)

AES_opt_key_schedule<NumKeys> uses the AES-NI–assisted "common-case" schedule via _mm_aesenclast_si128 (SubWord+RotWord) rather than AESKEYGENASSIST, following the 2015/751 trick that lets many keys share per-round constants. Round constants (aes.hpp:356-359):

con  = _mm_set_epi32(1,1,1,1)                            // initial Rcon, replicated
con2 = _mm_set_epi32(0x1b,0x1b,0x1b,0x1b)                // Rcon after GF(2^8) wrap (x^8 → 0x1b)
con3 = _mm_set_epi32(0x07060504,0x07060504,0x0ffffffff,0x0ffffffff)
mask = _mm_set_epi32(0x0c0f0e0d, …)                      // RotWord byte-shuffle

Per-round kernel detail::ks_rounds (aes.hpp:109-122):

block x2  = _mm_shuffle_epi8(key, mask);     // RotWord on the last word
block aux = _mm_aesenclast_si128(x2, con);   // SubWord (S-box) + XOR Rcon
…                                            // w[i] ^= w[i-1] cascade across the 4 words (prefix-XOR)
keys[i].rd_key[r] = _mm_xor_si128(aux, key);

The Rcon evolution matches FIPS-197 exactly: con doubles each round (rounds 1–8, aes.hpp:364-367); round 9 uses con2 = 0x1b and round 10 uses con2<<1 = 0x36 (aes.hpp:368-370) because Rcon has wrapped under the GF(2⁸) modulus x⁸+x⁴+x³+x+1. The <NumKeys> template loops ks_rounds over all keys per round, keeping constants in registers and exposing cross-key ILP. MITCCRH uses AES_opt_key_schedule<8>.

Implementation quality note

This is the high-quality fast formulation: aesenclast-based SubWord (avoids the slow AESKEYGENASSIST), a compact in-register prefix-XOR (a slli_epi64+pshufb variant — tighter than the canonical slli_si128 chain, aes.hpp:116-120), correct cheap Rcon evolution, and encryption-only (no wasted decryption schedule). The one identifiable headroom is the multi-key garbling hot path (MITCCRH re-schedules 8 keys every 8 gates): a VAES lane-parallel schedule could cut instruction count there. For the dominant PRP/PRG case the schedule is a tiny fraction of work and the scalar path is optimal.

Verified

The schedule is validated against the FIPS-197 Appendix C.1 known-answer vector and OpenSSL (test_aes.cpp).

ECB / ParaEnc & the tiled kernel

The core kernel aes_tiles_src<L,n_tiles> (aes.hpp:23-66) loads the 11 round keys into lane-broadcast vectors and runs textbook AES-128:

x = pt ^ rk[0];                          // initial AddRoundKey
for r in 1..9:  x = aesenc(x, rk[r]);    // 9 full rounds
x = aesenclast(x, rk[10]);               // final round, no MixColumns

ParaEnc<numKeys,numEncs> (aes.h:27-33) encrypts a K-major layout (keys[k] over blks[k*N … k*N+N)) and dispatches through ParaEnc_mem_tiles_impl (aes.hpp:184-211), splitting numEncs into compile-time tiles of widths {4,2,1} at the widest available SIMD tier. The "sweet spot" comment notes K·N ≤ 16 keeps all working blocks + 11 round keys in registers (aes.h:22-26).

The runtime-sized ParaEnc(blks, keys, K, N) (aes.hpp:392-407) drains each per-key stream with drain_tiles, peeling the widest tile first ({16,8,4,2,1} on VAES-512, else {8,4,2,1}, aes.hpp:234-242). A documented limitation: the runtime form preserves K-major layout and does not exploit cross-key ILP — hot callers that know (K,N) at compile time should use ParaEnc<K,N> (aes.h:35-40).

Also present: a Davies–Meyer counter-mode fill aes_ctr_fill_dm<W> (aes.hpp:343-348) and the secret-key detail::aes_ctr_fill<W> (aes.hpp:316-320); CTR plaintexts counter+i ⊕ tweak are built in-register (L::ctr_xor_tweak), so no counter buffer is written. A compile-time-zero tweak in the Plain path folds the per-counter XOR away (claimed ≈2× on single-block random_block on VAES-512, aes.hpp:286-289).

API / contracts

AES_opt_key_schedule<K>(user_key, keys) — pointers 16-byte aligned (asserted); fills keys[0..K). ParaEnc<K,N> — in-place K-major; out-of-place form uses __restrict__, so dst must equal src or be disjoint (aes.hpp:381-390). Convenience wrappers: AES_set_encrypt_key = AES_opt_key_schedule<1> (aes.hpp:409-411); AES_ecb_encrypt_blks = ParaEnc<1,N>.

5 · B · PRP — fixed-key random permutation

crypto/prp.h

PRP wraps a single AES_KEY and models f(x)=AES_K(x) as a random permutation when K is public (eprint 2013/426, prp.h:9-14). Its sole member is AES_KEY aes (prp.h:16). Constructors funnel to AES_set_encrypt_key; the const char* form reads exactly 16 bytes via _mm_loadu_si128, defaulting to zero_block when null (prp.h:19-27). permute_block(data, nblocks) (prp.h:29-38) encrypts in place, batching in chunks of AES_BATCH_SIZE = 64 via single-key runtime ECB; the chunking bounds register pressure per call.

Auditor scrutiny

Default zero key. The default key is zero_block (the fixed-key/RPM model). This is sound in the model but means every deployment with the default key shares the same permutation — confirm no protocol relies on key secrecy when using PRP/CCRH defaults. The const char* ctor does not check key length: the caller must guarantee ≥16 bytes (prp.h:18). Same alignment-assert caveat as AES (prp.h:30-31).

Verified

PRP ↔ OpenSSL ECB equivalence is tested across sizes straddling the 64-block boundary ({1,17,63,64,65,128,257}); re-keying by reassignment is tested (test_prp.cpp).

6 · B · PRG — AES-CTR pseudorandom generator

crypto/prg.h

PRG is AES-128 in counter mode with a secret key — the standard CTR-PRG, not the fixed-key correlation-robust construction. random_block calls detail::ParaCtrEnc using CtrStore::Plain, i.e. dst[i] = AES_key(counter+i) with no Davies–Meyer XOR-back and a secret key (the seed, prg.h:186-204). Security reduces to PRP/PRF security of AES-128 — a clean, standard assumption.

Seeding (prg.h:20-73)

  1. Given seed pointer → unaligned-safe _mm_loadu_si128 (prg.h:21-29).
  2. Else if test mode → deterministic counter seed (insecure by design, prg.h:32-37).
  3. Else with ENABLE_RDSEED → _rdseed64_step ×2, rejecting 0/all-ones to dodge an AMD CPU bug (PR #156), up to 10 attempts, falling back to urandom (prg.h:42-54).
  4. Else → /dev/urandom via std::random_device (prg.h:58-66).
Seeding caveats

rdseed failure silently degrades to urandom (fine for security — urandom is trusted — but an operator can't tell which source was used). The reseed(seed, id) id parameter is XORed into the low 64 bits (prg.h:67-73): PRG(seed, id) equals PRG(seed⊕id, 0), so id is a domain separator over a fixed seed, not independent entropy.

Streaming & forking

position()/seek(c)/fork_at(c) manage the 64-bit counter (prg.h:79-88). fork_at copies the PRG sharing the AES key (no re-key) and seeks.

CTR two-time-pad if forks overlap

Two forks at non-overlapping counter ranges are independent, but overlapping counter ranges across forks reuse keystream — a classic CTR two-time-pad if misused. The 64-bit counter can wrap after 2⁶⁴ blocks (unchecked, not a practical concern).

API surface: random_block (asserts 16-byte alignment, tiles in 64-block chunks, prg.h:186-204), random_data (requires 16-byte alignment, prg.h:91-100), random_data_unaligned (alignment-agnostic, uses memmove because the aligned region overlaps the destination — a refactor to memcpy would silently corrupt output, prg.h:135-184), random_bool (1 random bit per bool — 8× less AES work, prg.h:105-133), and operator() as a C++ UniformRandomBitGenerator backed by an alignas(block) buffer so the (block*) reinterpret meets the alignment assert (prg.h:206-225).

Security summary

Standard CTR-DRBG-style construction over secret-key AES-128. No reseeding-on-output-volume and no backtracking resistance: recovering the AES key + counter reproduces all past and future output — by design (it is a PRG, not a prediction-resistant DRBG), so seed compromise is total. Verified: counter continuity for N ∈ {1,7,16,17,64,65,257}, fork/seek reproduction, unaligned == aligned byte-and-counter parity (test_prg.cpp).

7 · B · F2K — GF(2¹²⁸) arithmetic

crypto/f2k.h crypto/f2k.hpp

Field & scope

The field is GF(2¹²⁸) under the GHASH polynomial x¹²⁸ + x⁷ + x² + x + 1 (f2k.h:14). Used for almost-universal hashing (VOLE/LPZK consistency checks), correlated-OT/VOLE arithmetic, and GF packing.

No GF(2⁶⁴) in these files

Despite the generic "f2k" name, only the 128-bit field reduction is implemented here. The per-64-bit-lane CLMUL is an implementation detail of the 128-bit multiply, not a GF(2⁶⁴) field op. A protocol needing a true 64-bit binary field will not find one in f2k.{h,hpp}.

mul128 — 128×128→256 carryless multiply (f2k.hpp:16-26)

The schoolbook decomposition, 4 CLMULs:

p00 = clmul_lo_lo(a,b)   // imm=0x00
p11 = clmul_hi_hi(a,b)   // imm=0x11
p10 = clmul_hi_lo(a,b)   // imm=0x10
p01 = clmul_lo_hi(a,b)   // imm=0x01
mid = p10 ^ p01
res1(lo) = p00 ^ (mid << 64)   // bslli_8(mid)
res2(hi) = p11 ^ (mid >> 64)   // bsrli_8(mid)

On aarch64 these map to vmull_p64 (simd_tier.h:286-312).

Documentation imprecision (and a measured non-result)

The function name historically referenced "Karatsuba", but the implementation is the 4-CLMUL schoolbook (Karatsuba would use 3 multiplies). This is the right choice: on hardware with ≥1/cycle PCLMULQDQ throughput (Haswell onward, Zen), Karatsuba's 3-CLMUL form trades a multiply for extra XORs and a longer dependency chain and yields no speedup for a single 128-bit product — a commonly observed non-result. The right description is "4-CLMUL schoolbook," not Karatsuba.

reduce / reduce_reflect — GF(2¹²⁸) reduction

reduce(lo,hi) (f2k.hpp:67-91) is the Intel CLMUL white-paper "Figure 7" shift-based reduction modulo x¹²⁸+x⁷+x²+x+1: the high half is folded via taps at bit positions 1, 2, 7. reduce_reflect (f2k.hpp:31-62) is the bit-reflected "GHASH form" (white-paper "Figure 5"), used when inputs/outputs are in reflected bit order. gfmul pairs with reduce; gfmul_reflect with reduce_reflect (f2k.hpp:96-109).

The two reductions are NOT interchangeable

They assume opposite bit orderings. Feeding reflected data to reduce (or vice versa) silently produces wrong field elements, and there is no type-level guard preventing it. The reduction constants are unnamed magic numbers; the clearest spec is the scalar reference ref_gfmul in test_f2k.cpp.

Inner products & deferred reduction

vector_inn_prdt_sum_no_red (f2k.hpp:176-193) accumulates the unreduced 256-bit product Σ aᵢ·bᵢ across SIMD lanes, then a single final reduce in the _red wrapper (f2k.hpp:198-203). Correctness rests on GF addition being XOR (linear): reduce(Σ unreduced(aᵢbᵢ)) = Σ reduce(aᵢbᵢ), so deferring reduction to one call is exact, replacing sz reductions with 1 (f2k.h:23-26).

Constant-time bool-selector path

When b[i] ∈ {0,1}, each term is a[i] & select_mask[b[i]] — a branchless masked XOR (f2k.hpp:211-223). The header advertises stable timing for adversarial b (f2k.h:37): no secret-dependent branch, no data-dependent CLMUL. This is the one timing guarantee directly relevant if b is secret. (a is still read in index order.)

Other routines: uni_hash_coeff_gen generates powers seed^(i+1) for polynomial hashing (f2k.hpp:230-261); GaloisFieldPacking::packing computes Σ data[i]·Xⁱ (single-bit polynomials ⇒ shift-and-accumulate, f2k.hpp:271-290), with a bool overload degenerating to LSB-first bit-packing (f2k.hpp:292-294); vector_self_xor is a 4-way XOR-reduction (GF sum, f2k.hpp:296-318).

Verified

gfmul is validated against a bit-level scalar reference and the field axioms (a·0=0, a·1=a, commutativity, distributivity); coeff-gen and packing against the gfmul reference (test_f2k.cpp).

8 · B · CCRH — circular correlation-robust hash

crypto/ccrh.h

CCRH implements H(x) = AES_K(σ(x)) ⊕ σ(x) — Davies–Meyer over the linear orthomorphism σ. It is the hash for free-XOR garbling and OT extension: correlation-robust even under the circular/related-key queries free-XOR induces. class CCRH : public PRP (ccrh.h:11) adds no members; default key zero_block (ccrh.h:12).

Single-block H applies σ then the Davies–Meyer "AES then XOR the AES input" variant (ccrh.h:15-19). The batched H<n> and runtime Hn (ccrh.h:28-46) fold σ into the tile source (AesMemXorSigmaTile, aes.hpp:144-154) so σ(src) stays live in-register as the XOR-back operand — single pass, no σ stash buffer.

σ is the load-bearing security component

Without σ, Davies–Meyer over free-XOR-correlated inputs is insecure. The raw ParaEncXorInput_impl building block (plain Davies–Meyer, no σ) is exposed in detail:: (aes.hpp:220-224) and is not circular-correlation-robust by itself. Any caller reaching into detail:: warrants scrutiny.

Auditor scrutiny

Out-of-place H<n>/Hn require out==in or disjoint — a shifted overlap is UB (ccrh.h:26-27,33). Large n in the compile-time H<n> spills registers; use Hn for big batches. Pinned to the exact definition H(x)=σ(x)⊕AES_K(σ(x)) across widths {1,2,3,4,7,8,15,16,17,31,33,64,65,1024} (test_ccrh.cpp).

9 · B · MITCCRH — tweakable multi-instance CCRH

crypto/mitccrh.h

MITCCRH<BatchSize> (default BatchSize=8) implements the tweakable circular-CR hash of Guo–Katz–Wang–Yang (eprint 2019/1168, mitccrh.h:8-11). Half-gates hashes each gate under a fresh tweak (the gate id); MITCCRH amortizes this by batch-scheduling BatchSize AES keys at once (one per tweak), then consuming them. This is the TMMO construction: H = σ(x) ⊕ AES_{K(tweak)}(σ(x)) — the tweak enters via the key, not the input.

State (mitccrh.h:15-19)

AES_KEY scheduled_key[BatchSize];   // pre-scheduled per-tweak keys
block   keys[BatchSize];            // raw per-tweak user keys (start_point ^ tweak)
int     key_used = BatchSize;       // cursor; == BatchSize means "exhausted, re-key"
block   start_point;                // the per-instance base S
uint64_t gid = 0;                   // running gate-id counter

setS(s) resets start_point, gid=0, and key_used=BatchSize (forces a re-key, dropping stale keys, mitccrh.h:21-25). renew_ks() derives keys[i] = start_point ⊕ makeBlock(gid++, 0), batch-schedules with AES_opt_key_schedule<BatchSize>, and resets the cursor (mitccrh.h:32-37). hash<K,H> (mitccrh.h:73-81) consumes K scheduled keys via ParaEncXorInput_impl; hash_cir<K,H> applies σ first, giving the full half-gates form (mitccrh.h:54-59).

Tweak-uniqueness is an unenforced caller obligation

The explicit-tweak renew_ks(tweaks) must not be mixed with the gid-based renew_ks() on the same instance — a tweak == makeBlock(g,0) for some gid g collides the AES key and breaks the security argument (mitccrh.h:44-46). The code does not enforce this. Likewise the caller must ensure gids never repeat within a start_point; re-setS resets gid to 0, so reusing the same S replays gids — safe only if S changed. This is the single most security-relevant misuse, and it is unchecked.

Auditor scrutiny

hash<K,H> static_asserts K ≤ BatchSize && BatchSize % K == 0 (mitccrh.h:75-76). Out-of-place forms allocate a stack array block tmp[K*H] (mitccrh.h:67) — very large K·H risks stack overflow (half-gates uses ≤8). Pinned against the explicit reference key derivation and hash_cir == hash(σ(·)) (test_mitccrh.cpp).

10 · B · Hash (SHA-256) & the random oracle

crypto/hash.h crypto/ro.h

A thin RAII wrapper over OpenSSL's EVP_MD_CTX running SHA-256; DIGEST_SIZE = 32 (hash.h:11-13). Copy is deleted (raw EVP_MD_CTX* would double-free, hash.h:24-25); used via std::optional<Hash> (emplace-only). Streaming: put/put_block → EVP_DigestUpdate; digest(out, reset_after) finalizes, where reset_after=false snapshots via EVP_MD_CTX_copy_ex and leaves the running transcript intact (hash.h:37-54) — the streaming Fiat–Shamir primitive. A thread_local persistent context backs the one-shot hash_once (hash.h:59-78).

hash_for_block — 128-bit truncation

hash_for_block returns the first 16 bytes of SHA-256 as a block (hash.h:82-86). Standard for a 128-bit label/key (pre-image/PRF target), but collision resistance drops to 64 bits (birthday). Any caller relying on 128-bit collision resistance would be under-protected — worth confirming none do. Cross-checked vs OpenSSL SHA256() and FIPS 180-4 KATs; snapshot non-disturbance verified (test_hash.cpp).

RO — random-oracle / Fiat–Shamir front-end

RO accumulates a typed, length-framed message M and squeezes a block, a 32-byte digest, or a curve point. The ctor takes a domain string and a mandatory block sid framed in immediately — every RO is session-bound (pass zero_block for a session-independent oracle, ro.h:36-39).

Injective framing (ro.h:42-83)

each field: u32_LE(type) || u32_LE(len) || bytes
type tags: kStr=1, kBytes=2, kBlock=3, kU64=4, kPoint=5     // ro.h:68-72

The type tag makes M injective in the call sequence (an absorb(block) differs from absorb(ptr,16) of the same bytes); length-framing prevents boundary-forgery ("ab"+"c" ≠ "a"+"bc"). Both pinned by an independent frame-builder in test_ro.cpp. Squeeze terminals (ro.h:53-64): squeeze_block (128-bit truncation caveat), squeeze_digest (full 32 B), squeeze_point (hash-to-curve with the domain reused as the RFC 9380 DST).

Auditor scrutiny

M is fully materialized in a std::vector before hashing (ro.h:74-85) — O(|M|) memory, a DoS surface if an attacker controls absorbed sizes. type/len are written in host byte order (raw memcpy), so cross-endian interop is not guaranteed. Domain separation is the caller's responsibility — distinct domain strings per usage; framing protects against accidental boundary collisions, not deliberate reuse of the same context.

11 · B · EC (P-256) & SessionID

crypto/ec.h crypto/ec.cpp crypto/session_id.h

OpenSSL EC_*/BN_*. Default curve P-256 (NID_X9_62_prime256v1, ec.h:85); curve NID is constructor-selectable, but hash-to-curve only supports P-256 (others error, ec.cpp:185-193). Three classes: Scalar (owns a BIGNUM; copy-and-swap assignment), Point (owns an EC_POINT; uncompressed 65-byte encoding), ECGroup (owns EC_GROUP + BN_CTX + cached order + a grow-only scratch buffer; non-copyable, non-movable, ec.h:83-124).

Point::from_bin — external-input validation (security-critical, ec.cpp:319-346)

EC_POINT_oct2point(...);                                  // rejects off-curve encodings
if (EC_POINT_is_on_curve(...) != 1) error("not on curve"); // explicit re-check  (ec.cpp:343)
if (EC_POINT_is_at_infinity(...)) error("point at infinity");   // ec.cpp:345
Subgroup safety is curve-dependent (cofactor-1 only)

The code reasons that on P-256 (cofactor 1), on-curve + non-identity ⇒ prime-order subgroup, so no separate subgroup check is needed (ec.cpp:339-342). This reasoning holds for cofactor-1 curves only. If the curve NID is ever changed to a cofactor-h>1 curve, from_bin would accept small-subgroup points — a real invalid-curve / small-subgroup attack surface. The guard is a comment, not an assert. Safe today because only P-256 is wired up.

Randomness & hash-to-curve

rand_scalar uses OpenSSL BN_rand_range in normal mode (the comment calls it "the only OpenSSL-randomness leak in the toolkit") and a PRG rejection sampler in test mode (ec.cpp:411-437). Production EC scalars depend on OpenSSL's RAND being properly seeded, independent of emp::PRG seeding. hash_to_point is a from-scratch RFC 9380 P256_XMD:SHA-256_SSWU_RO_ implementation (expand_message_xmd with length guards, count=2/L=48, SSWU with A=−3/Z=−10, ec.cpp:15-194), validated against the RFC 9380 §J.1.1 vectors (test_ecc.cpp).

Non-constant-time hash_to_point

The SSWU map uses BN_mod_sqrt (Tonelli–Shanks, not constant-time; fast path for P-256 since p ≡ 3 mod 4, ec.cpp:99-100,157) and branches on square-ness, plus BN_mod_inverse. Safe only if the hashed input is public. For the typical use (hash a public transcript to a CRS generator) inputs are public. Confirm no secret-dependent input ever reaches hash_to_point / RO::squeeze_point. On error() paths in-flight BIGNUMs leak, but process-fatal std::_Exit(1) makes that harmless.

SessionID (session_id.h:28-46)

A 128-bit session id + a child-derivation counter, giving each sub-protocol a distinct id without any value crossing the wire. derive() returns AES_{parent_sid}(counter++) (session_id.h:35-41) — two endpoints constructing children in the same order obtain identical ids deterministically (independent of test mode, by design).

Predictable if the seed is low-entropy

Derivation is a PRF only if the parent sid is high-entropy. A predictable seed (e.g. the zero_block default, session_id.h:44) yields predictable children AES_0(0), AES_0(1), …. Tolerable if the id is only a domain separator into a hash/RO; fatal if it is ever used as a key. The class makes no entropy guarantee — caller's responsibility. The counter is per-instance and never reset, so the determinism contract requires both endpoints construct sub-protocols in the same (data-independent) order.

12 · C · IO channels — IOChannel · NetIO · TLSIO · TraceIO

io/io_channel.h io/net_io_channel.h io/tcp_socket.h io/tls_io_channel.h io/trace_io.h

IOChannel — abstract base

A classic virtual-dispatch abstract base (not CRTP). Subclasses override two pure virtuals (send_data_internal / recv_data_internal, io_channel.h:80-81) plus optional flush/sync. All typed helpers (block/point/bool), counters, and Fiat–Shamir live in the base. Counters (send_counter, recv_counter, rounds = direction-change count, flushes_count) are advisory diagnostics (io_channel.h:37-51).

Point I/O wire format (io_channel.h:167-191)

send_pt: u32 length (host-endian) || uncompressed-point-bytes   // local len <= MAX_POINT_BYTES (assert)
recv_pt: u32 length || bytes ; hard len cap, then from_bin validates on-curve, rejects ∞
Peer length cap is always-on

The MAX_POINT_BYTES = 2048 cap is a hard error() check on recv before resize_scratch(len) or payload read. Local misuse checks, such as sending an oversized local point, remain debug assertions; the peer-controlled memory/DoS amplification path is not compiled out under NDEBUG.

Packed-bool I/O — the uninitialized-stack defense (io_channel.h:206-218)

if (batch % 8 != 0) buf[bytes - 1] = 0;   // zero the partial tail byte FIRST  (:212)
bools_to_bits(buf, data, batch);           // :213
Leak correctly closed

Because bools_to_bits preserves the high bits of the final partial byte (§A), an uninitialized stack buf would leak stack contents onto the wire and break FS determinism for non-multiple-of-8 lengths. The pre-zero closes that exact leak (whole bytes are fully overwritten). Tested with 1 MiB at offset +7 (test_netio.cpp).

Fiat–Shamir transcript (io_channel.h:95-143)

enable_fs(send_first) turns on two SHA-256 transcripts (fs_send_ absorbs sent bytes, fs_recv_ received). Exactly one party passes send_first=true. get_digest() returns the first 16 B of H(d_AB ‖ d_BA) with the two parties ordering their digests oppositely so both compute the same value, using non-destructive snapshots so the transcripts keep absorbing (callable repeatedly for sub-challenges). FS is off by default; enable_fs double-call is asserted (io_channel.h:96 — a release double-enable silently re-emplaces fresh transcripts).

NetIO & TCP

One TCP fd, full duplex (net_io_channel.h:37-152). Send side: a stdio "wb" FILE* with a 1 MiB stream buffer fronted by a 32 KiB coalescing buffer. Recv side: raw ::read() into a separate 32 KiB buffer, bypassing stdio. TCP_NODELAY is set (Nagle off). The recv path flushes pending sends first (net_io_channel.h:192) — critical, else a send-then-recv would deadlock with bytes still staged. The constructor fatal-errors on a bad port or fdopen failure (net_io_channel.h:82-83,139-150), consistent with the stack-wide error() model.

Thread-safety, make_sibling, no timeouts

NetIO is not thread-safe (unlocked coalescing) and non-copyable/non-movable. A debug-only touch_guard aborts on concurrent send/recv/flush (net_io_channel.h:58-79); in release the guard compiles away → silent wire corruption on misuse. make_sibling reconnects on the same role/port (net_io_channel.h:97-99) and has a documented ~1/5 same-port re-listen race (fixed downstream by a sync() before re-listen) — the burden is on the caller. No read timeouts: a peer that connects then goes silent hangs the process indefinitely (DoS vector). All transport errors → std::_Exit(1) (net_io_channel.h:202,228).

No peer authentication in plaintext NetIO

tcp::server_listen binds INADDR_ANY (all interfaces, IPv4-only) and accepts whoever connects — no peer authentication at the TCP layer (tcp_socket.h:22-47). client_connect uses inet_addr (no DNS; returns INADDR_NONE on malformed input without erroring, tcp_socket.h:52-73). For a plaintext deployment on an untrusted network, anyone who can reach the port can be the peer; the protocol's security must not assume an authenticated channel.

TLSIO — TLS 1.3 over TCP

Mirrors NetIO but the wire primitives are SSL_write_ex/SSL_read_ex (tls_io_channel.h:348-385). TLS version pinned to 1.3 (min and max, tls_io_channel.h:214-217) — no downgrade. Per-channel SSL_CTX (no global state). Config is caller-supplied PEM paths; require_peer_cert defaults true (mTLS, tls_io_channel.h:59), insecure_skip_verify defaults false (tls_io_channel.h:63). Verification loads the CA as trust anchor, sets SSL_VERIFY_PEER (+ FAIL_IF_NO_PEER_CERT for server mTLS, tls_io_channel.h:252-254), and re-checks SSL_get_verify_result() == X509_V_OK and peer-cert presence post-handshake (tls_io_channel.h:272-279). No custom verify callback (avoiding the "return 1 disables verification" footgun).

No hostname / identity verification

The code validates the chain to the CA and (for mTLS) that the peer presented a cert, but it never calls X509_VERIFY_PARAM_set1_host — so any certificate signed by the trusted CA is accepted as the peer. Fine for a private 2-party CA that only ever signs the two legitimate parties; a confused-deputy risk if the CA signs other entities. Document this as an assumption. insecure_skip_verify is an explicit dev escape hatch that must never be used in production. Constructing any TLSIO installs a process-global SIGPIPE ignore (tls_io_channel.h:71-79).

TraceIO — wire-byte record adapter

A wrapping IOChannel that tees every wire byte to <prefix>.send / <prefix>.recv while delegating to an underlying transport (borrowed, not owned, trace_io.h:28-83). Send teeing writes to the trace file before delivering (trace_io.h:56-66); recv delivers first then tees (trace_io.h:68-74). It calls the inner channel's *_internal directly so counters don't double-count. Short fwrite → abort().

Dev/CI tool only — plaintext wire bytes to disk

TraceIO writes plaintext wire bytes to disk (truncating fopen(…, "wb"), default umask, no encryption, trace_io.h:37-48). Protocol wire bytes are generally masked shares, but not always — this is an at-rest disclosure if sensitive. Its determinism contract ties correctness to the insecure test mode (§A). Never leave enabled in a deployment.

13 · D · Half-gate & privacy-free garbling

execution/half_gate.h execution/privacy_free.h

These are leaf primitives, not contexts — the only in-repo callers are the tests; downstream protocol repos (emp-ag2pc / emp-sh2pc) wrap them inside a block-valued BooleanContext (see Report 2). They compute the per-AND ciphertext (garble) and recover the per-AND output label (eval); XOR and NOT are free (a label XOR with the global delta), handled by the caller. Both assume the Free-XOR invariant: a wire's two labels differ by a global secret delta with LSB 1.

Half-gate (half_gate.h:14-43)

The hash is MITCCRH (§9): setS(delta) seeds it, hash_cir<K,H> applies σ then a batched Davies–Meyer AES fold. halfgates_garble (half_gate.h:14-29) emits a 2-ciphertext table from 4 hashes (hash_cir<2,2>); halfgates_eval (half_gate.h:31-43) recovers the output label from 2 hashes (hash_cir<2,1>). Branching is only on the public point-and-permute LSBs via the branch-free select_mask[pa]; the citation is Zahur–Rosulek–Evans 2014 (eprint 2014/756, half_gate.h:9).

Gate-id synchronization is a caller obligation

Both MITCCRH instances (garbler and evaluator) must be seeded with the same delta and advanced in lockstep gate-id-wise. A gate-id desync silently produces wrong labels. Correctness for all four input combinations is exhaustively verified (test_halfgate.cpp:29-45).

Privacy-free (privacy_free.h:14-45)

Used where the evaluator's input is allowed to leak (ZK / authenticated-garbling). Needs only one ciphertext per AND. privacy_free_garble (privacy_free.h:14-30) hashes only A's two labels with a per-gate tweak = 2·idx via AES_ecb_encrypt_blks + Davies–Meyer over σ, forces point-and-permute LSBs (&=0xfe / |=0x01), and outputs table[0] = H(LA0)^H(A1)^LB0. privacy_free_eval (privacy_free.h:32-45) branches on getLSB(A) — the structural reason the scheme is privacy-free (the evaluator's path depends on its own input bit).

Auditor scrutiny

The LSB manipulation writes *reinterpret_cast<char*>(&block) (byte 0, little-endian aliasing, privacy_free.h:24-25,40,43). idx is caller-supplied with no internal counter — garbler and evaluator must agree on idx per gate or labels diverge, and this is unchecked. By design this leaks the evaluator's wire value through the eval branch — confirm it is only used where that leakage is acceptable.

14 · SIMD-tier dispatch (cross-cutting)

core/simd_tier.h

Routing is centralized here. ISA tier macros are derived from compiler predefines (simd_tier.h:20-47):

EMP_HAS_AVX2    = __x86_64__ && __AVX2__        // simd_tier.h:20-24
EMP_HAS_AVX512F = __x86_64__ && __AVX512F__     //            :26-30
EMP_HAS_VAES    = __x86_64__ && __VAES__        //            :32-36
EMP_HAS_VAES512   = EMP_HAS_VAES && EMP_HAS_AVX512F          //  :45
EMP_HAS_VPCLMUL512/256  analogously                         //  :46-47

and assembled into per-module __attribute__((target(...))) packs (EMP_AES_TARGET_ATTR, EMP_F2K_TARGET_ATTR, simd_tier.h:49-66). AesLane<N> and ClmulLane<N> wrap AES-round / carryless-multiply primitives over N=1 (xmm), 2 (ymm, VAES+AVX2), 4 (zmm, VAES+AVX512) blocks (simd_tier.h:82-235); aarch64 has only N=1 (simd_tier.h:236-320). ctr_xor_tweak confirms the CTR counter is a 64-bit little-endian integer in the low 8 bytes of the block (simd_tier.h:94-96), relevant to PRG counter-collision reasoning.

Compile-time-only dispatch — no runtime CPUID

This is build-time dispatch via target attributes, not runtime CPU detection. No cpuid dispatch anywhere. A binary compiled with -mvaes -mavx512f executes VAES-512 unconditionally and will SIGILL on a CPU lacking those features — there is no fallback. Deployment hardware must match the build's -march. (The default build uses -march=native; pass -DEMP_TOOL_NATIVE_ARCH=OFF for a portable binary.)

aarch64 semantic adaptation

NEON's AESE XORs the round key before SubBytes (key-first), whereas AES-NI's aesenc XORs after (key-last). To stay x86-compatible the aarch64 path computes vaeseq_u8(s, zero) then XORs the real key separately, deliberately breaking the AESE/AESMC fusion the silicon wants (simd_tier.h:242-264). CLMUL maps to vmull_p64 (simd_tier.h:294-311). Cross-platform byte-identity rests entirely on the test suite's OpenSSL/reference cross-checks — there is no formal proof of x86↔aarch64 equality beyond those tests.
