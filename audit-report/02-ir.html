Boolean IR, contexts & sessions

The middle layer of the stack: a flat, protocol-neutral boolean-circuit IR with a native .empbc wire format, the C++20 BooleanContext concept and its reusable realizations, the generic WireValue structural-value contract, and the session layer that owns the I/O boundary. The central principle is one circuit source, many semantics: a circuit is written once over a BooleanContext and evaluated under any context — plaintext eval, gate counting, fingerprinting, IR recording, or a cryptographic backend — with the gate stream guaranteed identical across all of them. I/O (input/reveal) lives outside the gate concept, at the session.

A.1 · The IR data model

ir/program.h

One in-memory representation, one gate encoding, protocol-neutral — the IR knows nothing about parties, owners, argument boundaries, labels, or MACs (those live in the layers that produce and consume it). Opcodes (program.h:25):

enum class Op : uint8_t { And=0, Xor=1, Not=2, Const0=3, Const1=4 };  // == on-disk .empbc codes

Linear gates (Xor/Not) are free; And is the only costed op; public constants are first-class Const0/Const1 gates with no operands, so each backend realizes a known 0/1 wire however its protocol prescribes. A Gate is {in0,in1,out,op} with in1 normalized to 0 for Not/Const* (program.h:30-38). A BooleanProgram is a dense, SSA-like form (program.h:40-50):

uint32_t num_wires;     // total wire ids; valid ids [0, num_wires)
uint32_t num_inputs;    // input wires [0, num_inputs)
std::vector<Gate>     gates;     // topological (emission) order
std::vector<uint32_t> outputs;   // wire ids of the return value (may name inputs = passthrough)

Inputs are wires [0,num_inputs); every other wire is produced by exactly one gate in topological order; wire id == position, no holes.

A.2 · Validation invariants

ir/validate.h

validate_program (validate.h:19-79) is the single validation path — run at construct / load / before executing externally-supplied programs, never in the hot loop. It throws std::runtime_error on the first violation. Invariants:

  1. num_inputs ≤ num_wires (validate.h:20-22).
  2. Density: num_gates == num_wires − num_inputs — every non-input wire produced exactly once, no holes (validate.h:24-27). This is what lets num_wires size the scratch buffer exactly.
  3. Read-before-define / in-range reads in topological order (validate.h:39-43).
  4. Canonical operands: Not has in1==0 (validate.h:54-55); Const* have in0==in1==0 (validate.h:58-59).
  5. Output validity: each gate's out is in range, is not an input wire, and is defined exactly once (validate.h:64-71).
  6. Every output wire is in range and defined (validate.h:74-78).
Hardened against malformed headers

The defined_non_input bitmap is sized to num_wires − num_inputs (not num_wires), so a malformed file declaring a huge num_inputs cannot force a giant allocation during validation (validate.h:30-33). The dense-count check at validate.h:24-27 runs before that allocation. Each invariant is independently exercised in test_boolean_program.cpp:54-67, and the huge-sparse-header guard at test_boolean_program.cpp:121-130.

validate_program throws even in Release

Unlike most fail-fast paths in the tree (which call error() → exit(1)), the IR validator throws and is unconditional — it is not guarded by NDEBUG. A consumer that passes an externally-supplied program through validate_program (the .empbc loader does) gets a catchable exception in both Debug and Release, never silent corruption. This is the structural-safety baseline the rest of the layer relies on.

A.3 · The .empbc serialization format

ir/empbc.h

The one native persisted format. Layout (empbc.h:17-29): magic EMPB, version u16 (currently 1), index_width u8 (2 or 4), flags u8 (reserved, must be 0), then num_wires/num_inputs/num_outputs/num_gates as u32, then gate records, then outputs. Indices are stored 16- or 32-bit on disk (the writer auto-selects u16 when num_wires ≤ 0xFFFF, empbc.h:89) but always decode into a uint32_t BooleanProgram — the compact width is a disk detail, never a second in-memory IR.

Hostile-input-safe parser

A bounds-checked Reader (empbc.h:59-75) does fixed little-endian field reads (never a struct overlay / reinterpret_cast, so it is endian- and layout-independent). load_empbc (empbc.h:116-161) checks magic / version / index_width / reserved-flags, performs an overflow-checked exact size equation in 64-bit (empbc.h:137-141, rejecting both truncation and trailing bytes), range-checks each op code (empbc.h:150), and runs a final validate_program (empbc.h:159). The writer save_empbc validates first (empbc.h:88) so a malformed program can't be silently truncated by the u16 writer. The malformed/truncated/trailing/bad-magic/bad-op cases are all exercised in test_boolean_program.cpp:107-130.

Minor forward-compat ambiguity

The header flags byte must be 0 on read (empbc.h:126), but per-gate reserved padding bytes are read without value-checking (empbc.h:152-153) — a nonzero gate-record reserved byte is silently accepted. load_empbc_file reads the whole file into memory via ftell before the exact-size check (empbc.h:167-182).

A.4 · Analysis passes

ir/passes.h

Protocol-neutral pure-function analyses over a BooleanProgram, each computed once and optionally cached on the frontend Circuit. Per-wire scratch (first-def / last-use / level) uses signed vectors with a −1 "none" sentinel — local analysis state, never serialized (passes.h:8-12).

  • count_pass (passes.h:36-52) → AND/XOR/NOT/CONST tallies plus wire/gate/input/output counts (the stored-program analogue of CountCtx).
  • liveness_pass (passes.h:59-72) → per-wire last_use + first_def.
  • schedule_pass (passes.h:79-104) → AND-depth of each wire (Const=0; Not/Xor inherit max input level for free; And = 1 + max(level(in0),level(in1)), passes.h:86-93) and AND gates grouped by level — the input the scheduler consumes for round batching.
  • layout_pass (passes.h:118-148) → frees[gi] (slots recyclable after gate gi, with output wires excluded as roots) plus an output-rooted dead-code- elimination sizing (backward reachability counting reachable_wire / reachable_and, passes.h:130-146). Note this reports DCE sizes but does not rewrite the program — execution still walks all gates.

A.5 · Scheduled bulk replay

ir/schedule.h

Level-batched replay for a BulkBooleanContext: walk linear gates in topological order, but batch every ready AND layer (grouped by AND-depth) into one and_many call (schedule.h:77-99). Correctness is identical to the scalar execute_program; only batching changes. A ScheduledPlan (schedule.h:28-39) caches gate indices per AND-depth plus the program's identity (num_inputs/num_wires/num_gates/digest) for staleness checking. Per level: collect the level's AND operands/outputs → one and_many → run the level's linear gates in emission order.

Schedule digest guard is debug-only

Three staleness guards: the input count and the plan dimensions (num_inputs/num_wires/num_gates) are always-on error() checks (schedule.h:63-67), but the strong "same shape, different gate stream" guard plan.digest == digest_program(p) is wrapped in #ifndef NDEBUG (schedule.h:68-71). In a Release build, a ScheduledPlan whose dimensions match a different but same-shaped program would silently misexecute. Whether reachable depends on the backend's plan-caching discipline. (Scheduled == scalar replay is cross-checked on real AES/SHA/SHA3 circuits, test_builtin_circuits.cpp:48-55.)

A.6 · The single gate-walk & execution

ir/visit.h ir/execute.h

for_each_gate(p, visitor) (visit.h:23-34) is the only place the op-switch lives — adding a new Op is a compile error at every visitor, never a silent fallthrough. Execution has two layers:

  • Low level circuit::execute_program<Wire,Dispatcher> (execute.h:44-70) — seed input slots, walk once via for_each_gate through a Bridge that maps wire ids to value slots, copy outputs out; generic over a wire slot type and an in-place Dispatcher. Validates input/output counts (throws on mismatch, execute.h:50-53), sizes scratch to num_wires.
  • High level value-return bridge execute_program(ctx, program, inputs[, ws]) (execute.h:102-122) — the way stored .empbc builtins and compiled frontend circuits run through any BooleanContext, via a CtxReplayAdapter (execute.h:89-97) that turns the context's value-return gates into the in-place form. A ProgramWorkspace<Wire> (execute.h:79-86) holds reusable buffers (scratch, output, AND-batch buffers) — must be caller-owned or thread_local, never one process-wide mutable instance.
Trusts prior validation

execute_program validates input/output counts but re-checks nothing about the program's internal validity — it trusts validate_program ran at load/record time (the comment in validate.h:4-8 states this contract: validation runs at construction/load, "never inside the execution hot loop"). An externally-supplied, unvalidated program reaching execute_program would read out-of-range slots; the documented contract is that validation happens at the boundary, not here.

A.7 · Artifacts, assets & builtins

ir/artifact.h ir/assets.{h,cpp} ir/builtins.{h,cpp}

A CircuitArtifact is a BooleanProgram + a CircuitSignature (arg_widths + return_width — the argument boundaries the flat IR omits, artifact.h:21-33). validate_artifact (artifact.h:37-46) runs validate_program and checks sum(arg_widths) == num_inputs and return_width == outputs.size() in 64-bit. Big circuits (AES-128, SHA-256, SHA3-256, and the per-width float ops) ship as <name>.empbc under ir/files/ (63 committed assets) and are loaded/validated/cached once behind a mutex-guarded std::map (builtins.cpp:16-34, and float_circuit at builtins.cpp:36-56) then replayed through any context. The aes128 builtin has 256 inputs (pt ‖ key) and 128 outputs (test_builtin_circuits.cpp:63-64).

Asset loading: structurally safe, not authenticated — and EMP_CIRCUIT_DIR is honored first

Asset resolution searches, in order (assets.cpp:34-44): EMP_CIRCUIT_DIR (env var) → build-tree dir → install-tree dir. In a deployment where the environment is attacker-influenced, this allows redirecting builtin-circuit loads to an attacker-chosen directory. The .empbc loader's hostile-input safety is the only mitigation: a substituted file is still structurally validated, but it could be a different valid circuit (e.g. a weakened AES). There is no hash/signature/MAC on .empbc. Whether this matters depends on the trust model.

IR-replay path proven faithful

Each shipped builtin, replayed through ClearCtx (scalar and scheduled), is compared bit-for-bit against the context-native crypto kernel on random input (test_builtin_circuits.cpp:24-89): the test cross-checks aes128, sha256_256, and sha3_256_256 against the in-circuit kernels and asserts scalar-replay == kernel and scheduled-replay == scalar.

B.1 · The BooleanContext concept

ir/context/concept.h

The contract (concept.h:19-27):

template <class Ctx>
concept BooleanContext =
    std::regular<typename Ctx::Wire> &&
    requires(Ctx& c, typename Ctx::Wire a, typename Ctx::Wire b, bool v) {
        { c.public_bit(v) } -> std::same_as<typename Ctx::Wire>;
        { c.and_gate(a, b) } -> std::same_as<typename Ctx::Wire>;
        { c.xor_gate(a, b) } -> std::same_as<typename Ctx::Wire>;
        { c.not_gate(a) }    -> std::same_as<typename Ctx::Wire>;
    };
  • Ctx::Wire must be std::regular — copyable, because "fan-out needs copies" (concept.h:17-18). A wire is a cheap handle; the context holds the real state.
  • Gates return by value, not in-place mutation — static, inlineable dispatch (concept.h:4-7).
  • The concept does NOT require input/reveal/output. There is no private_input or reveal at the gate level — those live in the session layer. public_bit is the only "input" and it is for public constants.
  • No NOR/OR/MUX primitive — OR, equality, select are built in the value layer on the four primitives (e.g. a|b = a^b^(a&b)).

A refinement BulkBooleanContext adds and_many(o, a, b, n) to evaluate a whole AND layer at once (concept.h:32-37); the scheduler needs only this, and the win materializes only when a backend overrides the default scalar loop.

B.2 · The context taxonomy

Four reusable in-tree realizations, each a tiny struct static_assert-ed to satisfy BooleanContext, each giving a different interpretation of the same kernel — plus the block-valued garbling backends downstream protocol repos supply.

ClearCtx clear.huint8_t (0/1)Plaintext evalactual bitwise ops; no constant folding (so gate count matches a recorded program)
CountCtx count.huint8_tGate countingincrement counters; dedups the two constants to match Record

ClearCtx (clear.h:16-26): not_gate XORs only the low bit (a^1); the default and_many is a scalar loop. CountCtx (count.h:16-25) dedups Const0/Const1 with c0_seen/c1_seen so its tallies match a recorded program. RecordCtx (record.h:20-58) is the linchpin of "record once, replay anywhere": all inputs must be reserved before any gate — external_input(n) reserves [0,num_inputs), and a late external_input after the first gate hard-error()s (record.h:28-38; the inputs_closed flag is flipped by alloc_() at record.h:27, and the check is always-on, not debug-only). finish(outputs) runs validate_program (record.h:51-57), so a recorder cannot hand out malformed IR. DigestCtx (digest.h:20-51) mirrors RecordCtx's wire numbering and const-dedup precisely so the two digests agree.

B.3 · Cross-context checks & the digest duality

ir/context/checks.h ir/context/digest.h

Cross-context checks are debug-only by default

checks.h's check_same_context(l,r) (checks.h:23-34) verifies both operands' context pointers are non-null and equal. The default policy is DEBUG-ONLY (EMP_CONTEXT_CHECKS defaults to 1 only when NDEBUG is unset, checks.h:15-21); opt in to always-on with -DEMP_CONTEXT_CHECKS=1 (which raises error(), not just assert, so it survives NDEBUG once opted in), or force off with -DEMP_CONTEXT_CHECKS=0. In a Release build with no opt-in, mixing typed values from two different contexts is unchecked — most dangerous for id-based wires (two RecordCtxes both start ids at 0 and silently alias).

Digest encoder duality

DigestCtx and the standalone digest_program (digest.h:59-69) map ops to integers independently (the gate stream is hashed as And=0…Const1=4, with the same const-dedup and the same external_input(num_inputs) prefix mix at digest.h:62) — they MUST stay in sync (the header comment flags this at digest.h:8-9). Divergence would silently break determinism/staleness checks. digest_source (digest.h:75-83) streams a body through a DigestCtx and equals digest_program of the program the body records — the basis for the frontend's "recording is deterministic" test (test_circuit_fn.cpp:53-55). The digest is also the staleness guard on cached schedules (§A.5).

C · The generic WireValue concept

ir/wire_value.h

WireValue is the generic structural contract for a typed value object over a BooleanContext: a fixed-width bundle of wires with a clear-value codec, packable to / constructible from raw Ctx::Wire, rebindable to another context, and reporting its own context pointer. It is the context-free vocabulary the IR execution and session layers speak; it names no concrete value family (Bit/UInt/Int/Float/BitVec) and no protocol (wire_value.h:4-12). The contract (wire_value.h:26-52):

template <class V_>
concept WireValue =
    requires { typename V::Wire; typename V::context_type; typename V::clear_t; } &&
    BooleanContext<typename V::context_type> &&                         // a real gate context
    std::same_as<typename V::Wire, typename V::context_type::Wire> &&    // its wire IS that context's wire
    requires { typename std::integral_constant<int, V::width()>; } &&   // static, compile-time width
    std::same_as<typename V::rebind<V::context_type>, V> &&             // identity rebind
    requires(const V v, V::Wire* out)            { v.context(); v.pack_wires(out); } &&
    requires(V::context_type& c, const V::Wire* in){ { V::from_wires(c, in) } -> std::same_as<V>; } &&
    requires(V::clear_t cv, const bool* bits)    { { V::encode(cv) } -> convertible_to<std::vector<bool>>;
                                                   { V::decode(bits) } -> std::same_as<V::clear_t>; };

Two requirements distinguish it from a looser "has wires" check:

  • It binds to a real gate context. BooleanContext<context_type> and Wire == context_type::Wire are both required (wire_value.h:33-36) — a value cannot claim a wire type its context does not use.
  • It is fixed-width only. width() must be usable as a compile-time std::integral_constant<int, …> template argument (wire_value.h:37-39), so a runtime-width value (whose width() is requires (N>0) and absent at N==0) does not model WireValue. Session I/O and circuit compilation need a statically known width / signature; runtime-width values are in-circuit only (wire_value.h:14-17).

Every requirement is spelled through std::decay_t<V_> so the concept is stable under reference qualification. A convenience is_wire_value_v<V> variable template (wire_value.h:54) gives the boolean form. The frontend keeps a stricter RecordValue concept (circuit_fn.h:51-54) that refines WireValue by additionally requiring context_type == RecordCtx — see Report 3.

D.1 · The session layer & the reveal contract

ir/session/session.h ir/session/session_io.h

The gate concept has no I/O; the session concept is its peer and owns the I/O boundary. The session layer is a small family of concepts, not one monolith — a session names no circuit value family, because values are context-bound (UInt_T<DirectCtx,N> etc.), so adding a value family needs no session edit (session.h:4-18).

// ir/session/session.h
template <class S> concept Session =                         // the minimal protocol object
    requires(const S& s) { { s.party() } -> std::convertible_to<int>; };

template <class S> concept DirectSession =                   // + a direct/user gate context
    Session<S> && requires(S& s) {
        typename S::DirectCtx;
        requires BooleanContext<typename S::DirectCtx>;
        { s.direct_ctx() } -> std::same_as<typename S::DirectCtx&>;
    };

template <class S> concept CheckpointingSession =            // + a checkpoint() barrier
    Session<S> && requires(S& s) { s.checkpoint(); };
  • Session (session.h:26-28) is the floor: it reports a party() convertible to int. Nothing else is mandatory.
  • DirectSession (session.h:30-37) adds a DirectCtx type (itself a BooleanContext) reachable via direct_ctx() — the gate context operator-mode circuits are built over. A multipass protocol may own many internal pass contexts; DirectCtx is only the direct one.
  • CheckpointingSession (session.h:39-42) is an optional refinement for sessions carrying cross-call materialized state (direct / multi-pass protocols where wire labels must survive across invocations); it adds a checkpoint() barrier. A stateless session (e.g. ClearSession) has no such barrier and does not model this concept.

A spelling helper direct_ctx_t<S> = S::DirectCtx (session.h:47-48) lets tests/docs name a session's direct context without a session value alias.

The reveal contract is a separate, generic concept SessionIO<S,V> (session_io.h:18-29), parameterized over any WireValue V whose context_type is S::DirectCtx:

// ir/session/session_io.h — recipient-explicit, uniform over any WireValue
S::reveal_t<V> == std::optional<typename V::clear_t>   // required equality
s.input<V>(party, x) -> V                              // feed a clear value as `party`'s input
s.reveal(v, party)   -> std::optional<clear_t>         // present only on a party that learns it

Because SessionIO is generic over WireValue, a session names no value family — adding a value family needs no session edit (session_io.h:4-9). The revealed value is present only on a party that learns it (every party for PUBLIC, the named recipient otherwise) and std::nullopt elsewhere.

Why the session names no value family

Responsibilities are split so a session is value-family-agnostic: Session requires only party(); the gate context is named once as DirectSession::DirectCtx; and I/O is the orthogonal, value-generic SessionIO<S,V> over any WireValue. A new value family is usable through any session with no session edit, and CheckpointingSession is an opt-in refinement rather than part of the base contract.

D.2 · ClearSession — the plaintext reference

ir/session/clear_session.h

ClearSession (clear_session.h:39-101) is the trivial realization of the session contracts. It owns a pure ClearCtx (clear_session.h:100) and declares DirectCtx == ClearCtx (clear_session.h:41). Since there are no secrets in the clear, input becomes public wires and reveal reads wire values back into an always-populated optional.

  • direct_ctx() returns DirectCtx& (clear_session.h:52) — the gate context for value construction that is not I/O (public constants, crypto kernels).
  • party() reports PUBLIC (clear_session.h:48) — plaintext has no private party, so PUBLIC is the universal-knowledge convention.
  • reveal_t<V> is std::optional<typename V::clear_t> (clear_session.h:45); ClearSession always knows the value, so its optional is always populated, for every recipient.

Owner / recipient validation. input(owner, value) and reveal(value, recipient) accept and then ignore the party for routing (correct for plaintext — there are no secrets) — but only after validating it is a real party id: both error() unless the party is ALICE, BOB, or PUBLIC (clear_session.h:62-63 and clear_session.h:83-84; the enum is {XOR=-1, PUBLIC=0, ALICE=1, BOB=2} in runtime/core/constants.h). So ClearSession ignores routing but does not silently accept a garbage party id — a real protocol session (SH2PC / AG2PC) would additionally route by it, and an auditor should confirm those do not ignore it.

Codec-width check is always-on

input requires V::encode(value) to produce exactly V::width() bits — a mismatch error()s (clear_session.h:64-69), not a debug-only condition, because a codec that emits the wrong count would silently out-of-bounds the wire fill that follows. A static_assert additionally pins V::context_type == DirectCtx at both input and reveal (clear_session.h:60-61, clear_session.h:81-82). The bottom of the file static_asserts Session<ClearSession> and DirectSession<ClearSession> (clear_session.h:103-104); SessionIO<ClearSession,V> for a concrete value type is asserted in the emp-tool tests, since naming a value family in the header would make ir depend on circuits/.
