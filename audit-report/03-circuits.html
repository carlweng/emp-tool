Circuit value objects & frontend

The circuits layer: the context-bound boolean/garbled value families — Bit_T, BitVec_T, unsigned/signed integers, IEEE Float_T — the numeric kernels they build on, the data-oblivious sort, the in-circuit crypto objects (AES-128, SHA-256, Keccak/SHA-3), and the compile-once / run-anywhere frontend. Every value carries a pointer to its BooleanContext and forwards each gate to it — there is no global backend, so the same value code runs over plaintext eval, a counter, a digest, an IR recorder, or a real MPC backend.

A.1 · Foundations & the _T<Ctx> pattern

circuits.h typed.h value_traits.h

circuits/circuits.h and circuits/typed.h are pure umbrellas. Every value type is a class template whose first parameter is a BooleanContext:

template <BooleanContext Ctx>        class Bit_T;         // bit.h:19
template <BooleanContext Ctx, int N> class UInt_T;        // unsigned_int.h:31
template <BooleanContext Ctx, int N> class Int_T;         // signed_int.h:25
template <BooleanContext Ctx, int W> class Float_T;       // float.h:22
template <BooleanContext Ctx, int N> class BitVec_T;      // bitvec.h:20

Each exports a uniform contract: Wire/context_type/clear_t; an identity rebind<C2>; a static width() (fixed-width forms); pack_wires/from_wires (flatten / unflatten the wire layout); a context() accessor; and the clear codec encode/decode. value_traits<T> (value_traits.h:9-16) is a thin trait that forwards to these statics — the source of truth is the value type itself. This is exactly the metadata the compile()/run()/session machinery reads to learn a value's width and codec generically.

Default-constructed / moved-from values are UB to operate on

Every type has a defaulted ctor leaving ctx_ = nullptr and wires value-initialized but unusable (bit.h:27-31, with the same contract called out for UInt_T/Int_T/Float_T/BitVec_T). The class invariant — "a usable value has a non-null ctx_" — is caller-enforced; there is no release-mode guard. Cross-context mixing is guarded only by check_same_context, which is debug-only by default (see Report 2; ir/context/checks.h). The scalar numeric codecs cap at 64 bits: UInt_T/Int_T encode/decode static_assert(N ≤ 64) (unsigned_int.h:221, unsigned_int.h:225; signed_int.h:170, signed_int.h:175) because their host clear_t is uint64_t/int64_t. This is not a general width limit — BitVec_T<N>'s clear_t is std::array<bool,N> and round-trips at any width; for >64-bit values use BitVec_T for bit I/O, or convert through fixed widths.

A.2 · WireValue — the contract the families satisfy

ir/wire_value.h

The structural typed-value concept lives one layer down, in ir/wire_value.h, and is documented in Report 2. It is the context-free vocabulary the IR execution and session layers speak; it names no concrete value family and no protocol. A value type satisfies it through its own static members (wire_value.h:26-52):

template <class V_>
concept WireValue =
    requires { typename V::Wire; typename V::context_type; typename V::clear_t; } &&
    BooleanContext<typename V::context_type> &&                        // wire_value.h:34
    std::same_as<typename V::Wire, typename V::context_type::Wire> && // wire_value.h:35-36
    requires { typename std::integral_constant<int, V::width()>; } && // STATIC width() — fixed-width only
    std::same_as<typename V::template rebind<V::context_type>, V> &&  // rebind is identity
    requires(const V v, V::Wire* out) {
        { v.context() } -> std::convertible_to<typename V::context_type*>;
        v.pack_wires(out);
    } &&
    requires(V::context_type& c, const V::Wire* in) {
        { V::from_wires(c, in) } -> std::same_as<V>;
    } &&
    requires(V::clear_t cv, const bool* bits) {
        { V::encode(cv) } -> std::convertible_to<std::vector<bool>>;
        { V::decode(bits) } -> std::same_as<typename V::clear_t>;
    };

Every fixed-width family here models it: each exposes Wire = Ctx::Wire, a non-template static constexpr int width(), an identity rebind<C2>, pack_wires/from_wires, context(), and an encode/decode codec. Two structural requirements are worth flagging for an auditor: WireValue requires the value's context_type to itself satisfy BooleanContext and the value's Wire to be that context's Wire (wire_value.h:34-36) — so a value whose nested Wire drifts from its context's wire type is not a WireValue.

Runtime-width values are deliberately NOT a WireValue

WireValue requires a static, compile-time width() (wire_value.h:37-39). The runtime-width forms UInt_T<Ctx,runtime_width> / Int_T<Ctx,runtime_width> (N==0) expose only a non-static int width() const (unsigned_int.h:218, signed_int.h:167) and no encode/decode codec, so they do not model WireValue — they cannot be a compile() argument, a session input/reveal type, or any frontend boundary value. They are in-circuit only (see §B.2).

A.3 · Bit_T — the single garbled wire

bit.h

Holds one Ctx::Wire + a private Ctx* (reached via context()). Operators forward directly to context gates (bit.h:38-48):

a & b   → ctx_->and_gate(w, o.w)         // 1 AND
a ^ b   → ctx_->xor_gate(w, o.w)         // free
a | b   → a ^ b ^ (a & b)               // free-XOR-friendly OR: 1 AND
!a      → ctx_->not_gate(w)              // free
a == b  → !(a ^ b)  (XNOR)               // free
a.select(sel, t) → sel ? t : a   via kernel::mux
select argument order is inverted vs a C ternary

b.select(sel, t) returns sel ? t : b — the receiver is the false branch, t the true branch (bit.h:44-48; test_bit.cpp:23 is the ground truth: (b.select(a, t).w & 1) == (x ? 1 : y)). Consistent across all value types, but easy to get backwards.

A.4 · BitVec_T — packed bit vector

bitvec.h

std::array<Wire,N> with LSB at index 0; clear_t = std::array<bool,N>. The natural I/O type for crypto blocks and for assembling/disassembling wider values. operator[] returns a Bit_T view (bitvec.h:41); as_uint() reinterprets the same wires as UInt_T<Ctx,N> with zero gates (bitvec.h:44) — the bridge from crypto output into arithmetic. Bitwise ops are per-wire; equality returns a single Bit_T (bitvec.h:47-51). Shifts are pure wiring (no gates), zero-filling, and reject negative amounts via error() (bitvec.h:57-68); over-shift (s ≥ N) yields all-zero. Compile-time slice<Lo,Hi>() (half-open) and concat<M>(hi) (receiver = low half) at bitvec.h:70-80.

LSB-first is load-bearing

The index convention drives crypto byte layouts, codecs, and concat (low half = receiver). A single flipped index transposes results silently. BitVec shifts are always logical — arithmetic shift lives on Int_T.

B.1 · Numeric kernels — the auditable arithmetic core

numeric_kernels.h

All integer arithmetic delegates to bare-wire kernels in namespace emp::kernel, operating on Ctx::Wire* arrays directly, LSB-first throughout (numeric_kernels.h:1-13). Gate cost is measured in ANDs (XOR/NOT free).

B.2 · UInt_T — unsigned integer

unsigned_int.h

Storage is std::array<Wire,N> for fixed width (N>0) or std::vector<Wire> for runtime width (N==0), selected by std::conditional_t<is_dynamic,…> (unsigned_int.h:37-40); clear_t = uint64_t. Add/sub/mul are mod 2ᴺ (the kernels drop the top carry, unsigned_int.h:72-83). Comparisons return Bit_T (< via borrow-out; the rest derived, unsigned_int.h:85-90). Public-amount shifts/rotates are pure wiring (unsigned_int.h:100-113); secret-amount barrel shifts are log-depth (res = res.select(shamt[i], res << (1<<i))) with an overflow flag forcing the result to 0 when shamt ≥ N (unsigned_int.h:118-135). Bit-counting: popcount<R>, hamming_weight (safe width), leading_zeros (unsigned_int.h:139-154). Width views (slice/extract/concat/zext/trunc) are pure wiring (unsigned_int.h:173-194); as_signed() is a zero-gate reinterpret (unsigned_int.h:168-169, defined out-of-line in signed_int.h:193-200).

Edge cases

Constants beyond 64 bits zero-extend (carrier is uint64_t, unsigned_int.h:48-52). mod_exp assumes q ≠ 0 (precondition, unsigned_int.h:156-165). Raw popcount<R> silently truncates if R is too small (wrapping accumulator); the hamming_weight wrapper picks a safe width via kernel::bits_for(N).

Runtime-width integers — a first-class in-circuit feature

Beyond the fixed-width UInt_T<Ctx,N>, both integers support a runtime-width form selected by the width sentinel inline constexpr int runtime_width = 0; (unsigned_int.h:26): UInt_T<Ctx, runtime_width> and Int_T<Ctx, runtime_width> store their wires in a std::vector whose length is the width, fixed at construction (UInt_T(ctx, width), validated ≥ 1 at unsigned_int.h:45 and unsigned_int.h:232). The width-taking ctor and the conversions are gated requires (N == 0); fixed-width operations are gated requires (N > 0), and same_width_ enforces matching widths at runtime (unsigned_int.h:234-236).

ConversionDirectionNote
to_dynamic()fixed N>0 → runtimesame wires, zero gates (unsigned_int.h:196)

Runtime-width values are in-circuit only — NOT a WireValue

The N==0 form deliberately has no static width() and no encode/decode codec (the codec members are requires (N > 0)), so it does not satisfy the WireValue concept (ir/wire_value.h; see §A.2) and cannot be a compile() argument, a session input/reveal type, or a frontend boundary value. It is intended for inside a circuit body where a width is only known at run time (unsigned_int.h:4-12); cross to a fixed width via to_fixed<M>() before it reaches any I/O boundary. The signed form inherits the same property (signed_int.h:165).

B.3 · Int_T — two's-complement signed integer

signed_int.h

Shares the unsigned ripple kernels for add/sub/mul (low N bits are bit-identical, signed_int.h:61-62, signed_int.h:79). Differences:

  • Negation: -x = 0 − x (signed_int.h:63); −INT_MIN wraps to INT_MIN.
  • Signed comparison: signed_lt = (signs differ) ? sign(a) : unsigned_lt (signed_int.h:65-72).
  • Signed division: truncate toward zero, remainder takes the dividend's sign — implemented via magnitudes (ua = sa?−a:a) through unsigned div_full then sign-fix (signed_int.h:89-102).
  • Arithmetic right shift: public >> fills with the sign bit (signed_int.h:122-127); secret >> overflow → all-sign-bits (vs << overflow → 0, signed_int.h:132-148).
INT_MIN / −1 is a UB precondition

The magnitude trick negates INT_MIN to itself (doesn't fit), producing a wrong quotient — the test explicitly skips this case (test_int.cpp:100: if (ia == INT32_MIN && ib == -1) continue;). Division by zero inherits the unsigned saturating behavior. Flag any caller that can reach INT_MIN / −1. The MSB is always w[n_()−1]; an off-by-one in width corrupts sign handling everywhere, so the centralized n_() width helper (signed_int.h:183) is load-bearing.

C · Float_T — IEEE floating point

float.h float_traits.h

Represents IEEE-754 binary16 / binary32 / binary64 (W = 16/32/64), stored as std::array<Wire,W> with bit 0 = LSB and bit W−1 = sign (float.h:29). FloatTraits<W> provides per-width parameters and the host codec (float for fp16/fp32, double for fp64, float_traits.h:69-90); fp16 has no portable C++ scalar, so hand-written round-to-nearest-even converters (emp_float_to_half/emp_half_to_float, float_traits.h:19-67) handle subnormal/overflow.

Arithmetic is IR replay, not inline gates

The single most important architectural fact: float arithmetic replays a pre-recorded on-disk circuit fp<W>_<op>.empbc through the context (float.h:51-59: execute_program(*ctx_, circuit::float_circuit(W, op), …)). Binary ops (add/sub/mul/div/min/max), unary (square/sqrt/recip/rsqrt), ternary fma, and compare/classify (8-bit-result circuits → a single Bit_T, float.h:74-80). Only sign-bit ops are realized locally (no gates): abs forces the sign false, operator- flips it, copysign copies it, select is per-wire mux (float.h:113-119).

Trust extends to the off-tree assets

Float correctness depends on the on-disk .empbc assets in emp-tool/ir/files/ being the faithful circuits the tests assume — unlike the crypto builtins, the float circuits have no native-kernel counterpart and are validated only against the host FPU (test_float.cpp). circuit::float_circuit throws if an asset is missing. The replay_ return aliases a thread_local ProgramWorkspace valid only until the next replay (float.h:51-59); every public wrapper copies immediately (float.h:60-80) — confirm no path holds it across a second replay.

D · Sort — data-oblivious Batcher network

sort.h

A Batcher odd-even mergesort (not bitonic, sort.h:35-44). The compare-swap schedule is a fixed network, so the gate stream is identical for every input — independent of the data and of ascending (sort.h:7-8). That obliviousness is the security-relevant property: the comparison/swap sequence reveals nothing about the values. compare_swap computes both branches and conditionally muxes (no data-dependent branch, sort.h:17-23); the key+payload form applies the same conditional swap to the payload (sort.h:27-33). Generic over any value with operator< + select (UInt/Int/Float/BitVec-via-as_uint). The network works for any n, not just powers of two; complexity Θ(n·log²n) comparators, depth Θ(log²n).

Not stable; NaN float keys unordered

Batcher odd-even mergesort is not stable and makes no stability claim — two equal keys with distinct payloads can end up in either order. For Float_T, operator< uses the IEEE lt circuit, so NaN keys are unordered (compare false against everything) and their positions are unspecified. sort_by_key is named distinctly (sort.h:60-65) so an unqualified call over std::vector doesn't collide with std::sort via ADL.

E.1 · AES-128 circuit

circuits/crypto/aes128.h

All three crypto objects follow the same pattern: a detail:: layer on bare Ctx::Wire arrays + a thin public BitVec_T wrapper, LSB-first within each byte. The AES S-box is the Boyar–Peralta (2010) minimal-AND construction: 32 ANDs, 81 XORs, 4 NOTs (aes128.h:26-162). BP formulas are MSB-first, so indices are flipped once at entry (aes128.h:38) and once at exit (aes128.h:161) — pure renaming, no gates. xtime (GF(2⁸) ·x with poly x⁸+x⁴+x³+x+1) is pure XOR (aes128.h:166-177). The key schedule folds Rcon in with NOT gates (XOR-with-1 = NOT, aes128.h:211-213). State is column-major 4×4.

ComponentS-box callsTotal ANDs
S-box (Boyar–Peralta)—32 (asserted, test_crypto_aes.cpp:376)
Key schedule40 (4×10)1280
Block encrypt (10 rounds)160 (16×10)6400 = 200 S-boxes × 32 (test_crypto_aes.cpp:377)

CTR mode (NIST SP 800-38A) adds a public delta to the big-endian 64-bit counter in bytes 8–15 via a UInt_T<Ctx,64> add (carry past byte 8 dropped, aes128.h:310-323), allows a partial final block and a start_chunk offset (aes128.h:329-347), and is cross-checked against OpenSSL EVP_aes_128_ctr.

What guards correctness

The all-256-input S-box test + the explicit AND count (32) catch a transposed BP index or accidental extra gate. ShiftRows/MixColumns assume the column-major layout (aes128.h:242-289); a layout mismatch vs the I/O layout would produce a transposed-but-self-consistent cipher that still fails the FIPS-197 KAT. CTR drops carry past byte 8 — a stream > 2⁶⁴ blocks wraps silently (spec-conformant edge).

E.2 · SHA-256 circuit

circuits/crypto/sha256.h

Everything is a UInt_T<Ctx,32> op, so the audit reduces to "are these the FIPS-180-4 word operations." Round constants K[0..63] and IV are public constexpr tables (sha256.h:26-42). Logical functions use AND-saving forms: ch = z ^ (x & (y^z)), maj = (x&y) ^ (z & (x^y)) (sha256.h:46-51); Σ/σ are pure rotate/shift XORs (AND-free, sha256.h:52-59). The message schedule and 64 rounds are the standard recurrences mod 2³² (sha256.h:63-84). Message length N is compile-time public (static_assert N%8==0, sha256.h:99), so padding is public and the circuit shape is fixed per N; words are read big-endian via sha256_word_bit_index (sha256.h:88-90).

Byte-order crux + public length

The big-endian word indexing (3 − b/8 term, sha256.h:89) is the correctness crux; an off-by-one transposes bytes within a word — guarded by OpenSSL cross-checks at padding-boundary lengths {0,1,55,56,57,64,119,120}. SHA-256 here cannot hash a secret-length message. The fixed sha256(BitVec<256>) wrapper is 24744 ANDs (test_crypto_sha256.cpp:202).

E.3 · Keccak / SHA3-256 circuit

circuits/crypto/keccak.h

State is 25 lanes of 64 bits as UInt_T<Ctx,64> A[25] (FIPS-202 layout, lane (x,y) = A[5y+x], keccak.h:45). Keccak-f[1600] runs 24 rounds of θ → ρ/π → χ → ι (keccak.h:56-87). θ (column parities), ρ/π (lane rotate + index permutation), and ι (public-constant XOR = NOT gates) are AND-free; χ is the only nonlinear step — A[x,y] = row[x] ^ ((~row[x+1]) & row[x+2]), 1 AND per bit, computed row-by-row so the five lanes are snapshotted before update (keccak.h:77-82). SHA3-256 sponge: rate 1088, capacity 512, domain separator 0x06, pad10*1 (keccak.h:113-125); length N is compile-time public.

Transposition traps

The χ row snapshot is essential (overwriting in place would corrupt the x+1/x+2 reads — verified present, keccak.h:78-81). The ρ/π combined index map B[lane(y,(2x+3y)%5)] and RHO[y][x] (note [y][x], keccak.h:71-73) are the easy places to transpose; the lane-by-lane permutation test is the guard. The fixed sha3_256(BitVec<256>) wrapper is 38400 ANDs = 24 rounds × 25 lanes × 64 bits (one AND per χ output bit per round, test_crypto_keccak.cpp:238). Like SHA-256, message length is public — the AND count reveals the block count (expected for a public-shape circuit).

F · The portable circuit-function frontend

circuits/frontend/circuit_fn.h circuits/frontend/frontend.h circuits/frontend/rec.h

A user writes a pure circuit body once and runs it on any context. "Pure" means no input/reveal/OT inside a body — structurally impossible because the body is recorded through a RecordCtx, which has no I/O (circuit_fn.h:11-13). I/O is a session concern, around run(). Body forms: implicit-context ([](auto a, auto b){ return a+b; }) or explicit-context ([](auto& ctx, …){…}, required for nullary circuits and for making a constant with no anchor arg); a body callable in both forms is a contract error (circuit_fn.h:18-24).

RecordValue — the frontend's refinement of WireValue

The frontend speaks emp::WireValue generically and refines it for compilation. A RecordValue is simply a WireValue whose context_type is RecordCtx (circuit_fn.h:51-54):

template <class V_>
concept RecordValue =
    WireValue<V_> &&
    std::same_as<typename std::decay_t<V_>::context_type, RecordCtx>;

It is the only valid argument type to compile() (the body is recorded through a RecordCtx). The compiled Circuit<RetV, ArgVs...> class itself constrains its type parameters with RecordValue (circuit_fn.h:124-125), so only compile() / load helpers that normalize to rec:: types can name a Circuit. circuit_fn_traits + circuit_contract fire a specific static_assert per failure mode (non-circuit args, ambiguous body, reference/void/non-circuit return) at circuit_fn.h:95-113, each exercised by circuit_fn_contract_probes.cpp.

compile / run

compile<rec::UInt<32>, rec::UInt<32>>(body)                       // circuit_fn.h:190-222

Two positional-argument footguns

(1) Cross-context guards at run are debug-only (EMP_CONTEXT_CHECKS, circuit_fn.h:246-250): in release, passing an arg from a different context is unchecked — benign for value-only ClearCtx, the dominant correctness footgun for id-based/stateful contexts. (2) run validates total input width, not per-argument identity (circuit_fn.h:254-255): swapping two equal-width arguments produces a valid-but-wrong replay. Inherent to positional passing; mitigated because compile and run share the same ArgVs… order by construction. The argument-wire order depends on braced-init left-to-right evaluation (circuit_fn.h:202) — a refactor to a call-style aggregate would silently permute input wires.

rec.h provides recording aliases (rec::Bit, rec::UInt<N>, rec::Int<N>, rec::Float<W>, rec::BitVec<N>) = *_T<RecordCtx,…> (rec.h:7-11). There is also a no-compile frontend::run(body, args...) that evaluates a body directly on already-live typed values, recovering the context from the first argument (circuit_fn.h:267-288). test_circuit_fn.cpp compiles a 32-bit adder once, checks it is the size-optimal 31-AND adder, checks recording is deterministic via digest_program, and exercises explicit-context / nullary / fp32-replay / live-run / BitVec paths.

Where the session boundary actually lives

The frontend names no session and no party — I/O is the session layer's job (Report 2: ir/session/session.h + session_io.h). A DirectSession exposes its gate context via direct_ctx() and feeds/opens values via SessionIO<S,V>, generic over any WireValue (so adding a value family needs no session edit). ClearSession (ir/session/clear_session.h) is the plaintext reference: DirectCtx == ClearCtx, party() == PUBLIC, and its input/reveal validate that the owner/recipient is ALICE/BOB/PUBLIC before (correctly) ignoring routing in the clear; its reveal_t<V> is std::optional<clear_t>, always populated. The always-on codec-width check (V::encode must produce V::width() bits) is a codec bug guard, not a debug-only condition.
